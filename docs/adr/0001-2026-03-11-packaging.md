# [2026-03-11]

## Agent infrastructure should be unified in one repository with an installable tools package

### Context:

Agent infrastructure must serve multiple consumers — interactive Claude Code sessions, automated hooks, MCP tool calls, and Docker-based workflow pipelines — and should be distributable as a single coherent system. Several pre-existing components (agent-docs, code-analysis-tools, claude workflows) were developed independently but share dependencies and purpose.

Key concerns driving the architecture:

- CLI tools must be callable from hooks and MCP without environment activation
- Skills and agent definitions must be discoverable by Claude Code at standard paths
- Workflow pipelines depend on the same tools and agent definitions
- All components should be version-controlled together and installable in one step

### Decision

All agent infrastructure lives in a single repository (agent-infra) with the following structure: an installable Python package (agent_tools) exposing a CLI and MCP server, agent definitions symlinked into ~/.claude/agents/, skills symlinked into ~/.claude/skills/, and Docker-based workflow pipelines under workflows/. Installation is managed by a top-level Makefile using uv tool install for system-wide CLI availability.

### Consequences

**Good:**
- single install command sets up entire system
- CLI available system-wide without environment activation
- all components versioned and evolved together
- symlinks keep Claude Code discovery paths satisfied

**Bad:**
- repo encompasses heterogeneous concerns (docs, tools, workflows)
- uv tool install isolates package from project virtualenvs

**Alternatives considered — repository structure:**
- separate repos: components share dependencies and evolve together
- workflows separate: will share tools directly with agent_tools
- docs separate: adds indirection without benefit

**Alternatives considered — tools and MCP packaging:**
- separate CLI and MCP packages: duplication risk as tools diverge
- MCP only: CLI needed for hooks and direct terminal use
- CLI only: MCP needed for autonomous mid-task tool calls
- skills instead of MCP: skills require user invocation
- src/ layout: nesting without benefit for this use case

**Alternatives considered — installation:**
- Makefile only: loses Python entry points and dependency metadata
- pyproject.toml post-install hooks: fragile, cannot edit settings.json reliably
- shell script: harder to express named targets
- uv pip install: CLI unavailable outside active virtualenv
